# 项目面试指南

这份文档基于当前仓库源码整理，面向 `C++` 后端 / 服务端实习面试。文档只保留两大部分：项目介绍、面试问答。目标不是写开发日志，而是帮助你把这个项目讲清楚、讲准确、讲得有技术含量。

## 一、项目介绍

### 1. 项目定位

这个项目本质上是一个基于开源 `sylar` 二次开发的 `C++` 服务端系统项目。

它不是单纯的 AI API 转发器，也不是只做前端页面展示，而是围绕“AI 对话服务”这个场景，把服务端几个关键层次真正串起来：

- 底层是协程调度、事件循环、Hook、网络层
- 中间层是 HTTP 框架，包括路由、中间件、Session、SSE
- 上层是 AI 应用层，包括鉴权、模型路由、上下文管理、流式输出、异步持久化、消息队列和 RAG

如果一句话概括，可以这样说：

> 我这个项目是一个基于开源 sylar 做二次开发的 C++ 服务端项目。我在底层一方面参考 `libco` 增加了汇编版协程上下文切换，另一方面把原版的全局任务池、单 `epoll` 运行时，改成了“每个工作线程独立维护任务队列和事件循环、IO 操作受严格线程域约束、fd 具备线程亲和、连接接入按负载分配到目标 io 线程”的模型；再往上实现了 HTTP 框架和 AI 应用层，把鉴权、聊天、流式输出、异步持久化、RabbitMQ 和 RAG 串成了一条完整服务链路。

### 2. 整体架构

项目整体分成三层：

- `src/sylar`
  - 底层协程运行时与网络层
  - 负责 Fiber、Scheduler、IOManager、Timer、Hook、Socket、TcpServer 等基础设施
- `src/http`
  - HTTP 框架层
  - 负责请求解析、响应组装、路由、中间件、Session、SSL、SSE
- `src/ai`
  - AI 应用层
  - 负责鉴权、聊天服务、模型路由、持久化、消息队列、RAG

一条典型请求链路是：

```text
Client 发起 TCP 连接
  -> accept_worker 上的 TcpServer::startAccept
  -> 监听 socket accept 新连接
  -> 按 inflightClients + queuedTasks 选择目标 io 线程
  -> 把 client socket 投递到 io_worker 的指定 worker
  -> 绑定 client fd 到当前 io 线程，后续 IO 操作保持线程亲和
  -> HttpServer::handleClient
  -> 若启用 SSL，则包装为 SslSocket 并完成握手
  -> HttpSession::recvRequest（连接级解析状态，keep-alive 循环）
  -> Router / Middleware
  -> AiHttpHandlers
  -> ChatService
     -> LlmRouter
     -> OpenAI-compatible / Anthropic Client
     -> MessageSink
     -> RagRetriever / RagIndexer (optional)
  -> HttpResponse / SSE 写回连接
  -> 若 keep-alive 则继续收下一条请求
  -> 若客户端关闭、超时、出错或服务端主动结束，则连接断开
```

服务启动入口在 `src/ai/main.cc`。启动流程不是简单“起个 HTTP 服务”，而是会依次完成：

- 启用 Hook
- 加载并校验 YAML 配置
- 初始化 MySQL 连接池和仓储
- 选择持久化 sink：RabbitMQ 或本地异步 MySQL Writer
- 构建 Provider Client、Registry、Router
- 可选初始化 RAG 所需的 Embedding / Vector Store / Indexer
- 组装 `ChatService` 和 `AuthService`
- 创建 accept worker 和 io worker
- 注册请求 ID 中间件和鉴权中间件
- 注册 AI API 路由并启动 HTTP 服务

### 3. 底层协程与网络层改造

这部分不是“从零自研”，但也绝不是“直接拿原版 sylar 跑业务”。你真正做的是在原版思路上继续做运行时层的收敛和重构。

第一块改造是协程上下文切换。
原版使用 `ucontext` 做协程上下文切换，而你这里参考 `libco` 的思路，增加了汇编版协程上下文切换后端。核心思路是：协作式切换不需要保存完整通用上下文，只需要保存最小必要寄存器集合，所以可以让最热的切换路径更短。

第二块改造是调度器。
原版就是“全局任务池 + 全局锁”的模型，而你这里改成了每个工作线程维护自己的本地任务队列；同一个 `Scheduler` 内部的 worker 不再互相投递任务，只要已经运行在该调度器的 worker 线程上，`schedule(...)` 就直接回当前 worker 的 `localQueue`。跨线程收件箱仍然保留，但只用于外部线程或外部调度器向目标 worker 做远程投递，比如 `accept_worker` 把新连接交给 `io_worker`。收件箱又分成两层：热路径优先走 ring，满了再走兜底链表。这样做的目标，是尽量让调度热路径局部化，减少所有线程竞争同一个全局结构，也减少 worker 热路径上的额外判断。

第三块改造是 `IOManager`。
原版就是单 `epoll` 模型，而你这里改成了每个工作线程维护自己的 `epoll` 和唤醒句柄。最新这版实现里，`IO` 控制操作不再走“owner worker + 跨线程同步转发”，而是改成了更严格的线程域模型：当前线程必须就是这个 `IOManager` 的真实 worker 线程，同时对应 `fd` 还要通过 `FdCtx` 绑定到同一个线程；线程不对就直接拒绝操作。这样做的好处，是把 `fd -> 事件循环 -> 控制路径` 彻底绑死，避免跨线程改同一个 `epoll` 状态机。

第四块改造是定时器和等待语义。
定时器不再只是全局结构，而是跟 worker 更紧地绑定。Fiber 也不只是简单 `resume / yield`，而是引入了更显式的等待结果语义，比如 IO 就绪、超时、取消。这样上层 Hook 和 IO 层在恢复协程时，不只是知道协程该醒了，还知道它是因为什么醒的。

第五块改造是接入路径。
`TcpServer` 不再是 accept 到新连接后简单轮询扔给某个 io 线程，而是会先读取 `Scheduler` 的 worker 负载快照，再结合每个线程当前的连接数做打分，把新连接分到更合适的 io 线程；连接真正开始处理前，还会再次把 client fd 绑定到当前线程，保证单个连接生命周期里的线程亲和不被破坏。

整体上，这一层你可以概括成：

> 我不是把 sylar 当框架直接用，而是在它的协程和 IO 思路上，把运行时改造成了每个工作线程独立持有任务队列和事件循环、IO 操作受严格线程域约束、fd 保持线程亲和、连接接入按负载选目标 io 线程的模型。

### 4. HTTP 框架层

HTTP 层是你在底层运行时之上继续实现和扩展出来的部分，不是直接上第三方 Web 框架。

这一层主要包括：

- HTTP 请求解析和响应组装
- 路由系统
- 中间件机制
- Session 管理
- keep-alive 连接复用
- SSE 流式输出
- 可选 SSL 支持

路由上支持四类匹配：

- 精确匹配
- 参数路由
- 通配路由
- 默认兜底路由

路由优先级是清晰可解释的：精确优先，其次参数路由，再到通配和默认路由。这样面试里也比较好讲，因为它不是一堆模糊规则叠出来的。

中间件层面，AI 应用实际注册了两个关键中间件：

- `RequestIdMiddleware`
  - 负责生成或透传请求 ID，方便链路追踪
- `AuthMiddleware`
  - 负责 Bearer Token 校验，并把统一身份信息写入请求对象，供后续业务层读取

在服务模型上，HTTP 层不是把所有逻辑都塞进一个线程，而是明确做了 accept 和 IO 的职责拆分。`HttpServer` 运行在 `TcpServer` 基础上，accept worker 负责接收连接，io worker 负责请求处理和连接生命周期管理。对话流式接口则通过 SSE 实现，把上游增量结果按事件流持续推给客户端。

### 5. AI 应用层

AI 应用层是这个项目里最容易被低估、但其实最能体现你“懂 AI 服务端”的部分。它不是一个简单的模型转发层，而更像一个 AI 服务编排层。

这一层首先解决的是统一接入问题。
当前服务端已经支持两类上游客户端：

- OpenAI-compatible Client
- Anthropic Client

你不是把这两种接口直接暴露给业务层，而是先做成统一的 Client 抽象，再通过 `LlmRegistry + LlmRouter` 做模型路由。这样业务层并不需要知道“这次请求到底是走 OpenAI 协议还是 Anthropic 协议”，只需要表达自己的需求。

在路由策略上，当前实现不是写死的，而是有明确优先级：

1. 请求显式指定 `provider`
2. 如果没有显式指定，就按 `model -> provider` 映射
3. 如果还没有命中，就走默认 provider

这样设计的价值在于：

- 业务层不被单一厂商绑定死
- 可以支持不同模型归属不同 provider
- 后续切换供应商或增加新 provider 的改动范围更小

第二块是上下文管理。
`ChatService` 并不是收到一条消息就直接发给上游模型，而是会先做上下文准备，包括：

- 会话 ID 生成或复用
- 历史消息加载
- 上下文窗口裁剪
- 必要时的摘要刷新
- 可选的 RAG 召回内容拼接

这里真正体现“懂 AI”的地方在于：你意识到 AI 对话服务的难点不是只有网络调用，而是上下文预算和记忆组织。如果不做历史窗口控制和摘要压缩，消息会越来越长，成本和延迟都会失控。

第三块是流式输出。
AI 服务不只是普通的同步返回，还支持增量流式输出。这里你做的事情不是简单“把一个 stream 接口透传出来”，而是把上游增量事件流和下游 SSE 连接起来。上游模型客户端用流式回调持续产生 token / chunk，下游 HTTP 层通过 SSE 持续把事件送到客户端。这个设计说明你理解 AI 场景里“生成是连续过程，不一定非要等完整结果再返回”。

第四块是鉴权和会话隔离。
AI 应用层不是匿名聊天接口，而是完整带鉴权的服务。注册、登录、登出、查询身份这些接口之外，真正重要的是鉴权成功后会生成统一的主体标识 SID，后续聊天历史、上下文加载、RAG 检索过滤，都是围绕这个主体来做隔离。这说明你不是把 AI 服务当成单机 demo，而是按多用户服务的思路去设计。

第五块是持久化和解耦。
聊天消息不会在请求线程内同步直写数据库，而是先进入 `MessageSink`。当前支持两条路径：

- 本地异步 MySQL Writer
- RabbitMQ Message Sink，再由独立消费者写库

这样做的核心原因，是把“生成响应”和“消息落库”从同一条延迟路径里拆开，让请求主链路更专注于模型调用和结果返回。

第六块是可选 RAG。
你把 RAG 设计成可选增强能力，并明确拆成查询链路和索引链路，而且两条链路的执行方式并不一样：

- 查询链路：发生在主请求流程里。命中召回条件后，会先做 `query -> embedding -> 向量检索`，拿到结果后再继续组装 prompt 并调用大模型。从业务语义上看，这一步是同步等待检索结果返回；但底层 HTTP 请求接入了 `FiberCurlSession`，如果运行在 `IOManager/Fiber` 环境里，挂起的是当前 fiber，不是整个 worker 线程。
- 索引链路：主流程只负责把消息投递给 `RagIndexer`，真正的 embedding 和向量库写入由它内部的常驻后台线程按批异步执行，不是每条消息临时再开一个线程。

这种拆法比较成熟，因为它把“查询增强”和“异步索引”分成两个职责：查询链路直接服务当前回答，索引链路则放到后台慢慢做，不会把主链路逻辑搅在一起。

如果要一句话总结 AI 应用层，可以这样说：

> 我做的不是一个模型 API wrapper，而是一个 AI 服务编排层，解决了协议适配、模型路由、上下文管理、流式输出、异步持久化和可选 RAG 这些真正会出现在 AI 服务端里的问题。

### 6. 项目核心亮点

这个项目最值得讲的亮点，不是某一个点，而是它同时覆盖了底层运行时、中间协议层和上层 AI 服务编排。

可以归纳成下面几条：

- 基于开源 `sylar` 二次开发，不是直接套框架
- 参考 `libco` 增加汇编版协程上下文切换，压缩最热切换路径
- 把原版全局任务池、单 `epoll` 的调度和 IO 模型，改造成每个工作线程独立持有任务队列和事件循环、IO 操作受严格线程域约束、fd 保持线程亲和、连接接入按负载分配的模型
- 自己实现 HTTP 路由、中间件、Session、SSE，而不是直接上现成 Web 框架
- AI 应用层不只是调模型接口，而是完整做了协议适配、模型路由、上下文管理、流式输出、持久化和 RAG
- 把同步主链路和异步落地链路拆开，更贴近真实服务端设计

## 二、面试问答

### 1. sylar 网络层深挖问答

这一部分建议你按两档准备：

- 基础题
  - 先把协程、`epoll`、`Reactor`、`EAGAIN`、`Hook` 这些概念讲准确，保证术语层不会失分
- 进阶题
  - 再往你这份项目源码的实现细节落，包括 `Hook` 原理、调度器改造、严格线程域、连接接入、TLS 取舍、汇编上下文切换

#### 1.1 基础题

**Q：什么是协程？你会怎么用自己的话解释它？**

A：
我一般会把协程解释成“用户态的可暂停执行单元”。

它和普通函数最大的区别是：普通函数一旦调用，要么一直执行到返回，要么抛异常退出；但协程可以在执行过程中主动暂停，把控制权让出去，之后再从上次暂停的位置继续执行。所以协程本质上不是“另一个线程”，而是“在同一个线程里可切换的多个执行流”。

从运行时角度看，协程之所以能暂停再恢复，核心是它自己保存了一份执行现场，至少包括：

- 当前执行到哪条指令
- 当前栈顶在哪里
- 哪些关键寄存器要恢复

在这个项目里，对应的就是 `Fiber` 和底层 `Context`。切换时并不是让内核重新调度一个线程，而是在用户态把当前协程的现场保存下来，再把目标协程的现场恢复回 CPU。这样下次恢复时，它就能像“函数从来没中断过一样”继续往下跑。

所以协程最准确的关键词不是“并行”，而是“可挂起、可恢复、用户态切换”。它擅长解决的是高并发下大量等待场景，而不是替代 CPU 并行计算。

**Q：协程和线程到底有什么区别？为什么协程不能直接替代线程？**

A：
两者最本质的区别，在于调度者不同。

- 线程是内核调度的，操作系统决定什么时候切走、什么时候切回来；
- 协程是用户态调度的，只有在代码显式让出执行权时才切换。

这会带来一系列差异。

第一，线程有自己的内核调度实体，协程没有。多个协程通常只是跑在少量工作线程之上，所以协程本身不创造 CPU 并行，它只是把一个线程内的等待时间利用起来。

第二，线程切换通常更重，因为它会涉及内核调度、线程状态切换、可能的用户态/内核态来回切换；协程切换通常更轻，因为它主要是在用户态保存和恢复执行现场。

第三，线程天然可以利用多核，协程不能单独利用多核。要想同时跑满多核，底层仍然需要多个线程，然后在线程上复用大量协程。

所以更准确的说法不是“协程替代线程”，而是“线程负责承载和并行，协程负责在每个线程内部把等待型并发做薄”。这个项目本身也是这个思路：底层还是多 worker 线程，但每个 worker 线程里跑很多 Fiber。

**Q：为什么大家总说协程切换比线程切换轻？轻在哪里？**

A：
因为协程切换大多数时候只是在用户态切换执行现场，而线程切换通常要经过内核调度。

先看线程切换。线程是操作系统调度的，切换时通常至少会涉及：

- 当前线程运行状态变化
- 调度器选择下一个线程
- 保存/恢复更完整的执行现场
- 可能发生用户态/内核态切换

而协程切换，如果是像这个项目里的协作式有栈协程，切换点是明确的，运行时只需要保存最小必要上下文，比如 `rsp/rip` 和少量被调用者保存寄存器，再恢复目标协程的上下文即可。尤其你这里还把 `ucontext` 改成了汇编版上下文切换，热路径进一步缩短了。

所以“轻”主要体现在两点：

- 不依赖内核调度器介入
- 保存恢复的状态更少、更专用

当然，这里也要讲清楚边界：协程切换轻，不代表协程里的所有操作都轻。如果协程里真的做了阻塞系统调用，线程一样会被卡住。所以协程要发挥价值，必须配合 `Hook`、非阻塞 IO、事件循环一起看。

**Q：什么是有栈协程？什么是无栈协程？它们的区别是什么？**

A：
这个问题本质上是在问：协程暂停以后，执行现场到底怎么保存。

有栈协程的意思是，每个协程自己有一块独立栈空间。它暂停时，本地变量、函数调用链、返回地址这些状态天然就还留在自己的栈上；恢复时只要把 CPU 的执行现场切回那块栈，再从保存的指令位置继续跑就行了。

无栈协程则不是每个协程都持有一块独立调用栈，它更像“把异步流程拆成多个状态点”，暂停和恢复依赖的是编译器、框架或者业务代码显式维护状态，而不是保存一整套独立调用栈。

两者最大的区别可以从三点讲。

第一，表达能力不同。有栈协程更接近普通同步函数写法，因为它可以保留完整调用链；无栈协程往往更依赖状态机或者编译器生成状态。

第二，切换方式不同。有栈协程通常需要显式保存/恢复上下文和切换栈；无栈协程一般不需要切换独立栈，但会把“暂停点”编进状态机里。

第三，成本结构不同。有栈协程单个协程更重，因为每个协程要占一块栈；无栈协程单体更轻，但业务表达往往没那么自然。

你这个项目显然是有栈协程。因为这里有独立 `Fiber`、独立栈、独立 `Context`，还有 `resume/yield` 和汇编版上下文切换。面试时我一般会直接说：我这里做的是有栈协程运行时，不是基于回调机或者编译器状态机的无栈协程。

**Q：为什么你这里会选有栈协程，而不是无栈协程？**

A：
因为这个项目底层追求的就是“把网络并发重新写回顺序代码”，而这件事有栈协程更合适。

你这里的核心价值之一就是 `Hook`。业务层照样写 `accept/read/write/connect` 这种同步风格代码，底层在遇到 `EAGAIN` 时再去注册事件、挂起 Fiber、恢复后继续执行。要做到“恢复后还能像普通函数一样从原位置接着跑”，最自然的实现就是有栈协程。

如果用无栈协程，也不是不能做，但你通常要么依赖编译器协程，要么业务层接受更明显的状态机痕迹。这样对这个项目要表达的点反而不利，因为你这里想展示的是“底层运行时把异步 IO 语义收掉，上层尽量保持同步写法”。

所以我会把这个选择解释成一句话：

> 我选有栈协程，不是因为它绝对最好，而是因为它更适合把阻塞式网络 API 改写成协程化顺序代码，这和整个 sylar 网络层的目标是一致的。

**Q：`epoll` 在 sylar 这种协程网络层里到底扮演什么角色？**

A：
`epoll` 解决的是“哪些 fd 现在可以继续做 IO”这个问题，但它本身不负责协程切换。

很多人会把 `epoll` 和协程混在一起讲，其实它们分工不同：

- `epoll` 负责事件通知，告诉你哪个 fd 可读、可写、出错了；
- `Fiber` 负责执行流切换，决定当前跑哪个协程；
- `Hook` 负责把阻塞式 socket API 改写成“遇到 `EAGAIN` 就注册事件并挂起协程”。

在这个项目里，业务层写 `read/write/accept/connect` 这类看起来像阻塞调用的代码，真正的执行路径是：先试一次原始系统调用，如果不能立刻继续，就把 fd 事件挂到 `IOManager` 对应 worker 的 `epoll` 上，然后当前 Fiber `YieldToHold()` 让出去。等 `epoll` 告诉运行时这个 fd 已经 ready 了，再把那个 Fiber 调回来继续跑。

所以如果面试官问“`epoll` 的作用是什么”，比较准确的回答不是“它实现了协程”，而是：

> `epoll` 是这套网络层里的事件感知器，负责告诉运行时什么时候可以继续做 IO；真正把事件等待和顺序代码串起来的是 `Hook + IOManager + Fiber`。

**Q：什么是水平触发和边缘触发？为什么 `epoll` 常常会配 `ET`？**

A：
这两个概念的差别，本质上是“内核在什么条件下反复通知你这个 fd 还可用”。

水平触发，也就是 `LT`，可以理解成“只要这个 fd 还处于可读/可写状态，内核就会一直提醒你”。比如接收缓冲区里还有数据没读完，下次 `epoll_wait` 仍然会继续通知你这个 fd 可读。

边缘触发，也就是 `ET`，可以理解成“只在状态从不可用变成可用的那个边沿提醒你一次”。比如从“没数据可读”变成“有数据可读”时通知一次；如果你这次没把数据读干净，后面可能就不会重复提醒了，除非状态先回到不可读，再次变成可读。

所以两者的核心区别不是“谁更高级”，而是：

- `LT` 更宽松，更不容易写错
- `ET` 通知次数更少，但要求你每次被唤醒都尽量把数据处理到 `EAGAIN`

`epoll` 常配 `ET`，尤其在高并发网络层里，主要是因为它能减少重复通知和无效唤醒。对这种每秒可能处理大量 fd 事件的运行时来说，少一次重复事件通知，就少一层调度和遍历成本。

但代价也很明确：你必须把读写循环写对。也就是说，一旦收到读事件，通常要一直读到返回 `EAGAIN` 为止；写也是一样，要尽量写到不能再写。否则你可能因为没把缓冲区清空，而错过后续通知。

你这个项目的 `IOManager` 走的就是 `EPOLLET`，所以面试里如果被问到这一点，比较好的回答不是“因为大家都这么写”，而是：

> 我用 `ET` 是为了降低高并发下的重复事件通知成本，但它要求底层非阻塞 IO 和读写循环必须配合到位，通常要一直处理到 `EAGAIN`。

**Q：什么是用户态/内核态切换？它和这套协程网络层有什么关系？**

A：
用户态和内核态可以简单理解成 CPU 执行权限的两个层级。

- 用户态：业务代码运行的地方，权限受限，不能直接操作硬件和内核资源
- 内核态：操作系统内核运行的地方，可以管理调度、内存、网络、文件等底层资源

当程序调用系统调用，比如 `read`、`write`、`epoll_wait`、`accept` 时，CPU 会从用户态切到内核态，请内核帮你做事情，做完再切回来。这种来回切换本身就有成本。

这和协程网络层的关系在于：协程切换本身尽量留在用户态完成，但网络 IO 是否会把线程卡死，还是取决于你怎么和内核交互。

这个项目的思路不是“完全不进内核”，那不现实；而是：

- 真正需要和内核打交道的地方，交给 `epoll` 和非阻塞 socket
- 真正需要等待的时候，不让线程傻等在内核里，而是注册事件后先把当前 Fiber 挂起

这样做的结果是：线程不会因为某个 fd 暂时没数据就整体睡死，CPU 可以继续在同一个 worker 线程里跑别的 Fiber。也就是说，这套设计不是消灭内核，而是把“必须进入内核的等待”变成“进入内核登记事件 + 回到用户态继续调度别的协程”。

**Q：什么是 Reactor？用最直白的话怎么讲？**

A：
我会把 `Reactor` 解释成“事件到了以后，由应用自己去处理 IO 的模式”。

更直白一点说，就是内核只负责告诉你“这个 fd 现在可读了、可写了”，但真正的 `read/write/accept` 这些动作，还是应用层自己来做。像 `epoll` 这种机制，本质上就是 Reactor 体系里最常见的事件通知器。

放到你这个项目里，`IOManager` 就是在扮演 Reactor 运行时：

- 它先把 fd 事件挂到 `epoll`
- `epoll_wait` 回来后知道哪些 fd ready
- 然后恢复对应 Fiber 或执行对应回调
- 最后真正的 `read/write/accept/connect` 仍然是业务线程自己去做

所以 Reactor 的核心不是“有事件循环”，而是“内核通知就绪，应用自己完成 IO”。

**Q：那什么又是 Proactor？它和 Reactor 的差别到底在哪？**

A：
`Proactor` 可以理解成“内核或底层运行时把 IO 本身都做完了，应用拿到的是完成通知”。

也就是说，两者最大的区别不在“有没有事件”，而在“谁真正执行 IO”：

- Reactor：内核告诉你“可以读/写了”，应用自己再去读/写
- Proactor：底层先把读/写做完，应用拿到的是“已经完成”的结果

如果用一句很适合面试的话来概括，就是：

> Reactor 面向的是就绪事件，Proactor 面向的是完成事件。

你这个项目显然更接近 Reactor，而不是 Proactor。因为不管是原版 sylar 的 Hook 语义，还是你现在这版 `IOManager + Fiber`，本质上都是“先等 fd ready，再恢复协程，由协程自己继续调用原始 IO”。它不是底层帮你把数据都搬完了再通知你，而是恢复你继续把剩下的 IO 逻辑跑完。

**Q：那你这个项目具体是哪一种 Reactor 模型？为什么说是主从 Reactor？**

A：
你这个项目现在比较准确的说法，是“accept/io 分离的主从 Reactor”。

原因是它把“连接接入”和“连接处理”拆成了两层 Reactor 职责。

第一层是主 Reactor，也就是 `accept_worker`。它主要负责监听 socket、接受新连接，本身不负责跑完整业务处理链路。

第二层是从 Reactor，也就是 `io_worker` 下面那批 worker。新连接被 accept 之后，不是继续在 accept 线程里处理，而是按 `inflight * 8 + queuedTasks` 的分数选一个目标 io 线程，再把这个连接投递过去，由那个线程负责后续整个连接生命周期里的 IO、协议解析和业务处理。

所以它不是单 Reactor，因为 accept 和连接处理分开了；也不是那种所有从 Reactor 地位完全对等、接入端也能直接处理业务的混合写法，而是明显有一层只负责接入，一层负责处理。

这也是为什么“主从 Reactor”这个说法在你这里是成立的，而且不是套概念。因为源码里 `TcpServer` 明确要求 `accept_worker` 和 `io_worker` 是不同 `IOManager` 实例，连接接入后还要做目标线程选择和 fd 线程亲和绑定，这已经不是单纯“多线程 epoll”了，而是拓扑上确实分了主从角色。

**Q：为什么你这里不适合说成单 Reactor 或者简单的多线程 Reactor？**

A：
因为那样会把你这套拓扑说浅了。

如果说单 Reactor，意思通常是“同一套事件循环既负责 accept，也负责后续连接 IO 和业务处理”。这和你现在的代码明显不符，因为你这里 accept 路径和 handleClient 路径已经分到两个不同 `IOManager` 上了。

如果只说“多线程 Reactor”，虽然也不算错，但不够准确。因为多线程 Reactor 只强调“有多个线程处理事件”，没讲清楚这些线程之间的职责边界。而你现在这版最重要的恰恰就是边界很清楚：

- `accept_worker` 只负责接入
- `io_worker` 负责连接生命周期和请求处理
- 连接接入时还会按负载分配具体 io 线程
- 单个 fd 后续还要保持线程亲和

所以更准确的说法是：你这里不是泛泛的“多线程网络模型”，而是一个职责拆分明确、并且带连接感知分配和线程亲和约束的主从 Reactor。

**Q：什么是 `EAGAIN`？为什么它在这套协程网络层里这么关键？**

A：
`EAGAIN` 可以简单理解成：“现在这个操作还做不了，但这不是致命错误，你稍后再试”。

在非阻塞 socket 语义里，这个返回值非常常见。比如你去 `read`，但当前接收缓冲区里还没数据；或者你去 `write`，但发送缓冲区暂时写满了；这时候内核不会让线程傻等，而是直接返回 `-1`，同时把 `errno` 设成 `EAGAIN` 或 `EWOULDBLOCK`。

它之所以在这套协程网络层里关键，是因为它其实就是“同步写法”和“事件驱动语义”之间的转折点。

在你这个项目里，Hook 后的 `do_io(...)` 并不是一上来就挂起协程，而是先试一次原始系统调用。只有当结果是 `EAGAIN` 时，运行时才知道：当前这一步不能继续做了，但也不是硬错误，这时候最合理的动作不是报错退出，而是：

- 把这个 fd 需要的 `READ/WRITE` 事件挂到 `IOManager`
- 当前 Fiber `YieldToHold()`
- 等事件 ready 以后再恢复回来重试

也就是说，没有 `EAGAIN` 这个语义，Hook 很难优雅地知道“现在该挂起，而不是该失败”。它相当于给运行时提供了一个非常关键的信号：现在要从“直接执行”切换到“事件等待”模式了。

**Q：为什么你说 `EAGAIN` 是 Hook 机制里的关键分水岭？**

A：
因为 `Hook` 能不能把同步 API 改造成协程化等待，核心就在于它要区分三类结果：

- 成功：那就直接返回，不需要任何调度动作
- 真错误：那就直接返回错误，也不应该挂起协程
- 暂时不可继续：这时才应该进入“注册事件 + 挂起 + 恢复后重试”

而在非阻塞 IO 里，`EAGAIN` 恰好就是第三类结果的标准信号。

所以从实现上看，`do_io(...)` 的关键判断其实不是“这次 IO 失败了没有”，而是“这次失败是不是 `EAGAIN`”。如果不是 `EAGAIN`，说明要么是 `EINTR` 这种可立即重试场景，要么是真错误；只有 `EAGAIN` 才意味着这次失败对应的是一个可等待的 IO 条件。

所以我在面试里会这样概括：

> `EAGAIN` 对协程化 Hook 来说，不是一个普通错误码，而是告诉运行时“该把当前同步调用转换成事件等待了”的触发信号。

**Q：如果没有非阻塞 IO，只靠协程能不能解决网络并发问题？**

A：
不能，至少不能解决你这个项目想解决的问题。

因为协程只能决定“当前线程里跑哪个执行流”，但它不能阻止阻塞系统调用把整个线程卡死。假设你还是用阻塞 socket，然后某个协程在 `read` 上卡住了，那被卡住的不是“这一小段协程逻辑”，而是底层整个工作线程。这个线程上的其他 Fiber 也没法继续跑。

所以协程网络层真正成立的前提是三件事一起出现：

- socket 进入非阻塞语义
- 遇到 `EAGAIN` 时不报错，而是注册事件
- 当前 Fiber 挂起，让线程去跑别的 Fiber

缺了其中任何一环，协程都只是“代码结构变化”，不会真正变成高并发网络运行时。

这也是为什么我会说：协程不是魔法，真正让它在网络服务里产生价值的是“非阻塞 IO + 事件循环 + 挂起恢复语义”这整套配合。

**Q：为什么 `Hook + 非阻塞 IO + Fiber` 能把回调风格重新写成同步风格？**

A：
因为这套组合本质上是在把“本来要你手写的状态机”下沉到运行时里。

如果没有这套机制，你要写非阻塞网络程序，通常得自己处理一整套异步控制流：

- 这次 `read` 没读到怎么办
- 什么时机把 fd 挂到事件循环
- 什么时候恢复处理
- 恢复后这段逻辑要从哪里接着跑

这就是为什么传统非阻塞 IO 程序很容易写成回调地狱或者状态机代码。因为业务层自己必须记住“我做到哪一步了”。

而你这个项目的做法是把这件事收进运行时：

1. 业务层照样调用 `read/write/connect/accept`
2. Hook 先试一次原始系统调用
3. 如果直接成功，就像普通同步函数一样返回
4. 如果遇到 `EAGAIN`，就注册事件
5. 当前 Fiber 挂起，线程去跑别的 Fiber
6. 事件 ready 后，Fiber 被恢复
7. 代码从原来那一行后面继续执行

这里最关键的一点是：恢复时不是回到某个回调入口，而是回到“原函数原位置”。这就是为什么业务层看到的仍然像同步代码。

所以我会把这件事总结成一句话：

> 回调风格之所以能重新写回同步风格，不是因为阻塞消失了，而是因为等待被运行时接管了，业务层不再自己维护异步状态机。

**Q：那为什么不能只靠 Fiber，而一定要配合 Hook 和非阻塞 IO？**

A：
因为 Fiber 只解决“怎么切执行流”，不解决“系统调用会不会把线程卡死”。

假设没有 Hook，你在 Fiber 里直接调一个阻塞 `read`，那线程还是会被内核卡住；此时不是“当前 Fiber 暂停”，而是“整个 worker 线程暂停”。这个线程上的其他 Fiber 也都没机会运行。

再假设有 Fiber 但没有非阻塞 IO，那么就算你想在用户态切换，也拿不到 `EAGAIN` 这种“暂时不可继续、应转入事件等待”的信号，运行时也很难知道什么时候该挂起当前协程。

所以这三者的分工非常清楚：

- 非阻塞 IO 负责让系统调用在不能继续时立刻返回，而不是卡死线程
- Hook 负责把这种返回语义接管下来，改写成事件等待
- Fiber 负责把当前执行流挂起，并在事件就绪后恢复

少了任意一环，这套“看起来像同步代码、底层其实是事件驱动”的模型都不完整。

#### 1.2 进阶题

**Q：你觉得原版 sylar 网络层中的灵魂是什么？**

A：
如果只能挑一个最核心的点，我会说原版 `sylar` 网络层的灵魂就是 `Hook` 机制，而不是单纯的 `epoll` 或者 Fiber 本身。

原因是 `epoll` 只解决了“内核怎么告诉你 fd 就绪了”，Fiber 只解决了“用户态怎么切协程”，但真正把这两件事接起来、把业务代码从“回调式非阻塞编程”重新拉回“同步阻塞式写法”的，是 `Hook`。业务层照样写 `connect`、`accept`、`read`、`write` 这种看起来会阻塞线程的代码，但在 `Hook` 开启后，底层已经把它改造成“遇到 `EAGAIN` 就注册事件、挂起当前协程、等事件到了再恢复”的模型了。

所以我理解原版 `sylar` 最有价值的地方，不是它会不会调用 `epoll_wait`，而是它把 Linux 阻塞 socket API、`IOManager` 事件循环和用户态协程调度串成了一套完整语义。没有 `Hook`，上层就得自己写一堆回调和状态机；有了 `Hook`，上层才能用接近同步代码的方式写并发网络服务。

**Q：详细讲一下 sylar 的 Hook 机制是怎么实现的。**

A：
这套机制可以按五步理解。

第一步，是先把原始系统调用入口保存下来。`hook.cc` 里通过 `HOOK_FUN` 宏列出要接管的函数，比如 `sleep`、`socket`、`connect`、`accept`、`read`、`write`、`fcntl`、`close` 等，然后在 `hook_init()` 里用 `dlsym(RTLD_NEXT, "...")` 找到 libc 里的原始函数地址，保存到 `*_f` 这些函数指针里。这样后面即使我们自己定义了同名 `read`、`write`，也仍然可以回退去调用系统原始实现。

第二步，是用线程局部开关决定当前线程要不要启用 Hook。`hook.cc` 里有一个 `thread_local bool t_hook_enable`，通过 `set_hook_enable(true/false)` 控制。这样做的意义是：`Hook` 不是全进程无脑生效，而是只有运行在 `IOManager` 管理下的线程才进入协程化路径，普通线程仍然可以走原始阻塞调用。

第三步，是给每个 fd 建立运行时上下文。`FdMgr` 会为 socket fd 建立 `FdCtx`，里面记录这个 fd 是不是 socket、是不是已经关闭、系统层是否强制非阻塞、用户有没有显式设置 `O_NONBLOCK`，以及读写超时配置。这个对象很关键，因为 `Hook` 要不要接管，不是只看“当前函数是不是 `read`”，还要看这个 fd 当前处在什么语义下。

第四步，是把大部分阻塞 IO 收敛到 `do_io(...)` 这个模板函数。它的核心逻辑是：

1. 如果当前线程没开 Hook，直接调原始函数。
2. 如果 fd 没上下文、不是 socket、已经关闭、或者用户显式设置了非阻塞，也直接调原始函数。
3. 否则先试着调用一次原始 IO。
4. 如果成功返回，直接结束。
5. 如果被 `EINTR` 打断，就重试。
6. 如果不是 `EAGAIN`，说明是真错误，直接返回。
7. 如果是 `EAGAIN`，说明当前不能继续读/写，就调用 `IOManager::waitEvent(fd, READ/WRITE, timeout)` 注册事件并挂起当前 Fiber。
8. 等事件就绪、超时或取消后，Fiber 被恢复，再回到 `do_io` 循环里重新调用原始 IO。

这一步本质上就是把“阻塞等待”替换成“事件注册 + 协程挂起 + 恢复后重试”。

第五步，是把一些特殊系统调用单独处理掉。比如：

- `socket`：创建新 socket 后立即在 `FdMgr` 里建上下文。
- `accept`：本身也走 `do_io`，拿到新连接后再给新 fd 建上下文。
- `connect`：如果返回 `EINPROGRESS`，就等 `WRITE` 事件，再用 `getsockopt(SO_ERROR)` 判断连接是否真正成功。
- `fcntl`：拦截 `F_SETFL / F_GETFL`，把“用户显式非阻塞”和“系统内部非阻塞”这两个语义分开管理。
- `close`：在真正 `close` 前先 `cancelAll(fd)`，再把 `FdCtx` 从管理器中删掉，避免 fd 被内核复用后还命中旧上下文。

所以从源码角度看，`Hook` 机制不是简单重写几个系统调用，而是做了三层事情：函数拦截、fd 语义管理、等待语义改写。真正难的地方在后两层。

**Q：为什么说 Hook 机制本质上是在“同步写法”外面包了一层协程调度？**

A：
因为业务代码看到的还是同步接口，但运行时已经不是同步阻塞模型了。

比如你在业务里写一次 `recv(fd, buf, len, flags)`，从代码形态看，这个调用会一直卡住直到数据到来；但在 `sylar` 的实现里，它第一次调原始 `recv` 如果拿到的是 `EAGAIN`，不会让线程阻塞在那里，而是去 `waitEvent(fd, READ, timeout)`。`waitEvent` 内部会把当前 Fiber 记录到 `IOManager` 的事件上下文里，必要时还会挂一个超时定时器，然后执行 `Fiber::YieldToHold()` 把当前协程让出去。

这时候被挂起的是当前 Fiber，不是整个工作线程。工作线程可以继续跑其他 Fiber。等底层 `epoll` 发现这个 fd 可读，`IOManager` 就会在对应 `FdContext` 上触发事件，把等待结果写回当前 Fiber，再把这个 Fiber 投递回调度器。等它下次被调度起来，就从刚才 `YieldToHold()` 之后继续执行，重新再进一次 `recv`。

所以表面上你写的是阻塞式同步调用，实际上运行时语义已经变成：

`同步 API 形式 -> 非阻塞探测 -> 事件注册 -> 挂起当前 Fiber -> 事件到来恢复 -> 原地继续执行`

这就是为什么我说 `Hook` 的价值，不在“替换了多少系统调用”，而在它把编程模型从“显式状态机”重新拉回了“顺序代码”，同时又保留了协程调度能力。

**Q：原版 Hook 机制有没有边界？是不是所有网络库都能自动协程化？**

A：
肯定有边界，而且这个边界非常重要。`Hook` 不是魔法，它只能接管它看得见、也适合接管的阻塞路径。

第一种边界，是用户自己显式把 fd 设成了非阻塞。`fcntl(F_SETFL, O_NONBLOCK)` 被拦截后，`FdCtx` 会把 `userNonblock` 记下来。后面 `do_io(...)` 一旦看到 `ctx->getUserNonblock()` 为真，就直接调用原始系统调用，不再走协程化路径。因为这种场景下，用户已经明确选择了非阻塞语义，运行时不能再擅自替他改回“伪同步”。

第二种边界，是这个 fd 不是 socket，或者当前线程根本不在 `IOManager` 上下文里。比如普通文件描述符、或者某个后台线程没有跑 `IOManager`，那就算开了 Hook，也没有可挂起的事件循环和协程调度器，这时候只能退回原始阻塞调用。

第三种边界，是某些第三方库内部的等待模型根本不走 `sylar` 这套路径。最典型的例子就是 `libcurl easy`。这个仓库里后面专门实现了 `FiberCurlSession`，就是因为单靠 `sylar hook` 并不能把 `curl_easy_perform` 自动协程化。原因有两个：

1. `libcurl` 会主动把 socket 设成 `O_NONBLOCK`，这样 `do_io(...)` 入口就会因为 `userNonblock=true` 直接 bypass。
2. `libcurl easy` 内部等待 IO 就绪时会用 `select/poll`，而这两个函数并不在 `sylar` 当前的 Hook 列表里，所以线程还是会直接阻塞在内核里。

所以比较准确的说法是：`sylar` 的 Hook 很强，但它接管的是“自己定义好的阻塞 socket 调用边界”，不是所有第三方网络库都能无条件吃到这套协程化能力。

**Q：为什么你会想改造 sylar 网络层的 N:M 调度逻辑？**

A：
我最直接、也最真实的动机，就是我觉得原版“全局任务池 + 全局大锁”的调度模型在高并发下性能不会好。

因为原版那种做法的核心问题不是“代码风格老不老”，而是所有 worker 都去竞争同一个全局任务结构。任务一多，锁竞争会直接落在最热路径上。你可以把它理解成：协程切换再轻、Hook 再优雅，只要所有线程最后都要抢同一把锁，吞吐和尾延迟都会被这个共享热点拖住。

而且这个问题不是只停留在理论上。结合我后面写的优化文档来看，原版除了全局任务池的大锁问题，还叠加了共享 `epoll`、共享唤醒路径、跨线程控制语义过松这些热点。也就是说，调度层和 IO 层都偏共享结构，线程数一上来，热点会被进一步放大。

所以我后面做的事情，本质上不是“为了改而改”，而是想把运行时从“强共享结构”收敛到“每个 worker 尽量局部化持有自己的任务和事件资源”的方向上。这是一个很典型的服务端运行时优化思路。

**Q：描述一下你改进后的 sylar 网络层调度逻辑。**

A：
我现在这版调度逻辑，核心可以概括成一句话：不再让所有线程竞争一个全局任务池，而是让每个 worker 优先处理自己的本地任务；同一 `Scheduler` 内部 worker 不允许互投，只有外部线程或外部调度器向目标 worker 交任务时，才会走远程收件箱。

具体来说，第一层是本地队列。每个 worker 都有自己的 `localQueue`，如果任务就是投给当前 worker，或者调度器判断本地执行最合适，就直接进这个本地队列，路径最短。

第二层是跨线程收件箱。每个 worker 都有自己的 mailbox，而且不是单一结构，而是两层：

- 热路径走 `mailboxRing`，它是有界 ring buffer，用来处理高频远程投递；
- ring 满了以后，再退到 `mailboxFallback` 这个无界链表，保证极端情况下也不丢任务。

这样做的意思很明确：性能好的快路径和容量兜底的慢路径分开设计，既不想让外部远程投递都走昂贵结构，也不想为了快路径把语义做成“ring 满就丢任务”。

这里有一个这版代码里很重要的收敛点：只要当前线程已经是这个 `Scheduler` 的 worker，`schedule(...)` 就会直接把任务压回当前 worker 的本地队列，不再做 worker 之间的互投，也不再保留单独的内部控制投递通道。换句话说，同一调度器内部现在就是“本地队列优先且唯一”，远程投递只留给外部线程或外部调度器进入这个运行时的场景。

第三层是 worker 选择逻辑。现在 `selectWorker()` 不再是简单轮询，而是用了 `P2C`，也就是从两个候选 worker 里选 `queuedTasks` 更小的那个。这里它主要服务的是外部线程或外部调度器进入目标 worker 的场景，比如 `accept_worker` 接到新连接后，把处理任务投给 `io_worker` 的某个目标线程。因为纯 RR 在短连接和突发场景下很容易继续把慢 worker 打满，而 `P2C` 成本很低，但负载均衡效果明显更好。

第四层是唤醒逻辑。跨线程投递不是每次都无脑写 `eventfd`，而是只有目标 worker 当前确实处于 `sleeping=true` 的时候，才 `tickle(worker)` 把它唤醒。这样可以减少高并发下的唤醒风暴。

所以如果用一句面试话术来总结，我会这么说：

> 我把原版全局任务池的 N:M 调度模型，改成了每个 worker 自己维护本地任务队列；同一 `Scheduler` 内部 worker 不再互投，当前 worker 内部调度直接回本地队列；外部线程和 `accept_worker` 向目标 worker 投递时，再通过 `P2C` 选 worker 并走 `mailboxRing + mailboxFallback`，唤醒也只在目标 worker 睡眠时发生。核心目标就是把热路径局部化，减少全局共享结构带来的锁竞争和唤醒热点。

**Q：改进后的 IOManager 为什么改成了严格线程域 + fd 线程亲和，而不是继续用 owner worker 跨线程转发？**

A：
因为“多开几个 `epoll`”并不等于真正解决了并发问题。真正麻烦的是：一个 fd 到底归谁管，谁有权修改它的事件状态。你如果还允许其他线程随时把 `addEvent/delEvent/cancelEvent` 这类操作转发过去，本质上还是在维护一条跨线程控制路径，只是共享热点从“单 epoll”变成了“多 epoll + 同步转发”。

我现在这版实现把这个问题收得更死。第一层约束，是 `IOManager` 自己先检查当前调用线程是不是“这个 `IOManager` 的真实 worker 线程”。源码里 `getCurrentWorkerForOp()` 会同时校验两件事：当前 `Scheduler::GetThis()` 必须就是这个 `IOManager`，而且 `getWorkerThreadId(worker)` 必须等于当前真实线程 id。只要不满足，就直接返回错误，甚至不会进入后面的 `epoll_ctl` 路径。

第二层约束，是 `FdCtx` 里新增了线程亲和字段。`ensureFdAffinity()` 会从 `FdMgr` 里取出对应 `FdCtx`，然后调用 `bindAffinityIfUnset(current_tid)`。如果这个 fd 还没绑定线程，就把它绑到当前线程；如果已经绑过，而且不是当前线程，就直接返回 `EXDEV`。也就是说，`IOManager` 不是“看到 fd 就能操作”，而是还要求这个 fd 的线程亲和和当前线程严格一致。

这套做法和旧的 `owner worker` 思路相比，有两个本质区别。

第一，它不再依赖跨线程同步转发。以前那种“当前线程不是 owner，就投给 owner 线程做，再同步等结果”的模式，虽然也能维持语义正确，但控制路径还是跨线程的，复杂度和同步成本都在。现在这版是非法线程直接拒绝，你必须在正确线程里做正确的 fd 操作。

第二，它把 `fd -> thread -> epoll` 的关系变成了硬约束，而不是运行时尽量维护的软约束。这样事件上下文、超时定时器、取消语义、恢复路径都天然落在同一个线程域里，状态机不会在多个 worker 之间来回撕扯。

这也是为什么我现在更愿意把它描述成“严格线程域 + fd 线程亲和”，而不是“多 worker epoll”。因为真正有价值的不是 epoll 数量，而是你有没有把控制权边界讲清楚。

**Q：为什么你还要改 `TcpServer` 的连接接入逻辑，而不是 accept 到新连接以后简单轮询分给某个 io 线程？**

A：
因为 accept/io 分离只解决了“接入线程和处理线程分开”，但没有解决“新连接具体该分给哪个 io 线程”这个问题。如果这里只做简单轮询，那么某个 io 线程可能已经挂着很多长连接或者慢连接了，结果新连接还是被机械地继续往它身上塞，最终会放大局部热点。

我现在这版 `TcpServer` 会先向 `io_worker` 取一份 `getWorkerStatsSnapshot()`，里面至少有每个 worker 的 `queuedTasks` 和 `sleeping` 状态；同时 `TcpServer` 自己还维护了一份 `inflightClientsByThread`，记录每个线程当前正在处理多少连接。接到新连接后，不是直接 RR，而是先算一个分数，大体上是 `score = inflight * 8 + queuedTasks`，然后选分数最小的线程；如果并列，再做轮转打散。

这里有两个实现细节很值得讲。

第一，为什么 `inflight` 权重比 `queuedTasks` 大？因为对连接分配来说，存量连接数往往比瞬时任务队列长度更能代表一个 io 线程的持续压力，所以代码里把它乘了更高权重。

第二，连接被投到目标线程后，不是立刻就算完了。真正进入 `handleClient()` 之前，还会先给 client fd 做一次 `FdCtx::bindAffinityIfUnset(current_tid)`，确保这个连接整个生命周期里的 IO 操作都落在同一个线程域里；等连接结束后，再把 `inflight` 计数减回去。

所以这块改动的本质，不是“换了一个更花哨的轮询算法”，而是把连接接入真正和后端 io 线程的负载状态绑定起来了。这样 accept 路径和 IO 路径的语义才是闭环的。

**Q：为什么你这版 `IOManager` 在线程不对时会直接返回 `EXDEV`，而不是偷偷帮调用方转发到正确线程执行？**

A：
因为这不是“图省事”，而是刻意选择的一种 fail-close 语义。

如果线程不对时还悄悄帮调用方把 `addEvent/delEvent/cancelEvent/cancelAll` 转发到正确线程，你表面上看是在提高易用性，实际上是在继续保留一条隐式跨线程控制路径。这样会带来两个问题。

第一，调用方会误以为“在哪个线程操作 fd 都行”。一旦这种认知扩散开，线程域边界就会越来越模糊，最后系统虽然还能跑，但你很难再从架构上保证状态机一定落在同一个线程里。

第二，问题会被延迟暴露。现在这版实现里，线程不对就直接 `EXDEV`，还会打日志告警，等于是在最早的边界把错误暴露出来；如果偷偷转发，很多本来应该在开发阶段暴露的线程域违例，会变成线上才出现的时序问题、尾延迟问题或者调试困难问题。

所以我这里的设计选择很明确：线程域不是“建议遵守”，而是“必须遵守”。不满足约束时，宁可立刻失败，也不做隐式修复。这种策略对底层运行时更健康，因为它让调用约束和代码行为保持一致。

**Q：为什么 fd 线程亲和是放在 `FdCtx` 里做，而不是只放在 `IOManager::FdContext` 里做？**

A：
因为 `IOManager::FdContext` 只覆盖“这个 fd 当前在 epoll 上注册了什么事件”这一个层面，但线程亲和约束影响的范围更大，它不只是 epoll 事件管理问题。

`FdCtx` 属于 `FdMgr`，是整个运行时里更基础的 fd 运行时上下文。它本身就已经在管“这个 fd 是不是 socket、是否关闭、系统非阻塞、用户非阻塞、超时配置”这些跨模块语义。所以把线程亲和也放到 `FdCtx` 里，等于把它定义成“fd 的基础属性”，而不是某个 `IOManager` 私有状态。

这样做的价值有三点。

第一，Hook 层、Socket 层、SSL 层、IOManager 层都可以共享同一份 fd 线程归属信息，而不是每层自己再维护一份。

第二，亲和约束会跟着 fd 生命周期走，而不是只跟某次事件注册走。也就是说，就算当前这个 fd 暂时没挂 epoll 事件，它仍然有自己的线程归属语义。

第三，这对 SSL 包装场景尤其重要。`SslSocket::FromSocket()` 是先 `dup` 一个 fd，再把它接管成新的 socket 对象。如果线程亲和只存在 `IOManager::FdContext` 里，那这个新 fd 很可能根本还没进入 epoll 事件层，自然也拿不到正确的线程归属信息；但把亲和放在 `FdCtx` 里，就可以在更早阶段先把语义立住。

所以我会把这个设计理解成：线程亲和不是 `IOManager` 附带的实现细节，而是 fd 运行时模型本身的一部分。

**Q：为什么 `SslSocket::FromSocket()` 里要主动创建 `FdCtx` 并绑定线程亲和？这一步不做会有什么问题？**

A：
因为 `SslSocket::FromSocket()` 走的不是普通新建 socket 路径，而是“拿一个已有 socket，`dup` 出新 fd，再包装成 SSL socket”这条路径。对这种路径来说，很多底层运行时假设不能天然成立，必须手动补。

第一，`dup` 出来的 fd 不一定已经在 `FdMgr` 里注册。尤其你代码里还专门兼容了某些变体场景，所以这里不能假设“只要有 fd，就一定已经有对应 `FdCtx`”。如果不主动 `get(fd, true)` 创建运行时上下文，后续 `Socket::init(fd)` 或更底层的 Hook/IO 逻辑就可能拿不到完整 fd 语义。

第二，SSL 包装后的这个 fd 仍然要遵守你整套底层网络层的线程域约束。也就是说，它不是“因为进了 SSL 层，就可以脱离原来的 fd 亲和模型”。所以这里会在包装时先做一次 `bindAffinityIfUnset(current_tid)`；如果发现这个 fd 已经绑定到了别的线程，就直接失败，而不是继续带着错误线程归属往下跑。

第三，当前 SSL 路径还特意兼容了 `SSL_ERROR_SYSCALL + EAGAIN/EWOULDBLOCK/EINTR`，按可重试处理。这说明 SSL 层并不是一个脱离底层运行时的黑盒，而是在主动适配非阻塞 + Hook + 线程域约束这套组合语义。

所以这一步最本质的作用，是把“已有 socket 包装成 SSL socket”这条特殊路径重新拉回到统一的 fd 运行时模型里。你可以把它理解成：SSL 不是另起一套规则，而是继续服从底层 fd 生命周期、线程亲和和错误语义这套约束。

**Q：为什么 TLS 路径现在保留的是重试语义，而不是把 `WANT_READ/WANT_WRITE` 改成 `waitEvent` 后再继续？**

A：
因为我确实做过这个实验，但当前代码没有保留，原因不是“不会做”，而是我用压测结果证明了这条路在当前口径下不划算。

理论上，把 `SSL_ERROR_WANT_READ/WANT_WRITE` 改成“注册事件、挂起 fiber、等可读可写再继续”是很诱人的，因为它看起来更像纯粹的协程化等待语义。但真实问题在于，TLS 不是普通明文 socket 读写，它前面还叠着 OpenSSL 自己的状态机、握手阶段、记录层处理和用户态缓冲。你把它改成 `waitEvent` 之后，不一定真的减少了尾延迟，反而可能引入更多调度切换和状态推进成本。

至少从你现在这份优化复盘文档看，当时在高并发短压口径下，这个实验版本没有带来整体收益，甚至不少场景下 HTTPS p99 更差，所以最后回滚了。

因此当前实现保留的是更保守的策略：

- `WANT_READ/WANT_WRITE` 继续重试；
- `SSL_ERROR_SYSCALL + EAGAIN/EWOULDBLOCK/EINTR` 也按可重试处理；
- 真正重点放在 SSL 包装路径的 fd 上下文一致性和线程亲和约束上。

面试里我会这样讲这件事：我不是机械地追求“所有阻塞点都协程让出”，而是做过实验、看过指标，最后保留了当前更稳的实现。这个回答比空讲原理更有说服力，因为它体现的是工程取舍，而不是概念正确。

**Q：为什么你要给 Fiber 增加显式等待结果，也就是 `WAIT_READY / WAIT_TIMEOUT / WAIT_CANCELLED` 这一套？**

A：
因为在 IO 等待场景里，“协程被唤醒了”这个信息本身是不够的。你还得知道它是为什么被唤醒。

比如一个 Fiber 在等 `READ` 事件，它恢复回来可能有三种原因：

- 真正读就绪了；
- 超时定时器到了；
- fd 被关闭或者事件被主动取消了。

如果没有显式等待结果，上层只能知道“我从 `YieldToHold()` 回来了”，但没法区分到底是 ready、timeout 还是 cancelled，这样超时处理、错误码设置、清理逻辑都会变得含糊。

所以我这里给每个 Fiber 增加了 `waitToken` 和 `WaitResult` 语义。等待前先 `beginWait()` 生成 token；事件上下文保存这个 token；等 `triggerEvent()` 时，把 `WAIT_READY / WAIT_TIMEOUT / WAIT_CANCELLED` 写回对应 Fiber；等 Fiber 恢复以后，再通过 `consumeWaitResult(token)` 读取结果。这样恢复语义就是显式的，而不是靠猜。

这个设计还有一个隐含收益，就是它能防止旧等待结果污染新等待。因为 wait token 是递增的，如果某个旧事件晚到了，但 token 对不上，Fiber 就不会错误消费旧状态。也就是说，这套机制不仅让语义更清楚，也让竞态下的行为更安全。

**Q：为什么你要仿造 libco 实现汇编上下文切换？它的原理是什么？和 ucontext 的区别在哪里？**

A：
先说动机。因为 `ucontext` 太通用了，而协作式协程切换其实并不需要那么完整的上下文语义。

`ucontext` 这套东西的定位，本来就是通用用户态上下文机制。它要处理的上下文更完整，通常包括寄存器现场、栈信息、信号掩码等内容，所以功能很全，但代价就是切换路径更重。对高频协作式协程切换来说，这个成本不划算。

而我这里参考 `libco` 做的汇编版切换，本质上是专门为“有栈、协作式、用户态切换”这个场景裁剪过的。当前 `Context` 结构里只保留了最小必要寄存器集合：

- `rsp`
- `rip`
- `rbx`
- `rbp`
- `r12 ~ r15`

这些正好对应 x86_64 调用约定里被调用者保存寄存器和最关键的控制流寄存器。对协作式切换来说，保存这些就够了。

它的汇编原理其实很直接：

第一步，切出当前协程时，把当前栈顶和返回地址保存下来。汇编里先取 `8(%rsp)` 作为切回时应该恢复的 `rsp`，再把当前栈顶保存的返回地址取出来写到 `from->rip`。同时把 `rbx`、`rbp`、`r12-r15` 这些寄存器也写进 `from` 上下文。

第二步，切入目标协程时，从 `to` 上下文里把上述寄存器恢复出来，再把 `to->rsp` 写回 CPU 的 `rsp`，最后把 `to->rip` 取出来直接 `jmp *%rax` 跳过去。注意这里不是普通函数调用，而是直接跳到目标上下文记录的指令地址执行。

第三步，首次切入一个新 Fiber 时，不是从某个“被保存过的现场”恢复，而是靠初始化时手工布置栈。`InitChildContext()` 会先算好独立协程栈顶，做 16 字节对齐，再在栈顶预留一个返回地址槽位，写入一个保护函数 `FiberEntryReturnAbort`，然后把 `ctx.rsp` 指向这个槽位，把 `ctx.rip` 指向真正的协程入口。这样第一次切进去时，就像 CPU 本来就在这个独立栈上执行这个入口函数一样。

和 `ucontext` 相比，它们的区别可以从三点讲：

第一，保存现场的范围不同。`ucontext` 更全，汇编版更小；所以汇编版热路径更短。

第二，语义目标不同。`ucontext` 是通用机制，汇编版是“只服务当前协程模型”的专用实现，所以前者更重但更通用，后者更轻但更平台相关。

第三，工程 tradeoff 不同。汇编版性能更好，但可移植性和维护成本也更差，因为你要自己对 ABI、栈对齐、寄存器保存规则负责；`ucontext` 则简单很多，但切换开销高。

所以我在面试里一般会把这件事总结成一句话：

> 我仿造 `libco` 上了一套手写汇编上下文切换，不是为了炫技，而是因为协作式协程切换只需要最小寄存器集合，不需要 `ucontext` 那么重的通用语义。这样能把最热的切换路径显著压短，但代价是平台相关性更强、实现和维护难度更高。

### 2. HTTP 框架层深挖问答

**Q：为什么你没有直接用现成 Web 框架，而是自己实现这一层 HTTP 框架？**

A：
因为这个项目里，HTTP 层不是一个可有可无的薄壳，而是整个运行时和业务层之间的中间协议层。如果直接上现成框架，很多真正有含量的东西就被吃掉了，比如连接级 keep-alive 循环、路由优先级、中间件和业务处理的执行语义、Session 生命周期、SSE 流式输出，以及这些机制怎么和底层 `sylar` 的协程调度、accept/io 分离模型拼起来。

换句话说，我这里不是为了“重复造轮子”而自己写 HTTP，而是因为这个项目本身就在展示服务端分层能力。底层 `sylar` 解决的是协程和 IO 事件循环，AI 应用层解决的是鉴权、聊天、RAG 和持久化，而 HTTP 层负责把请求解析、路由分发、链路拦截、连接复用和流式协议这几个问题收拢成一套一致语义。如果这层直接交给现成框架，你就只能讲“我会用框架”，很难讲“我怎么定义这层的行为边界”。

而且从实现上看，这层并不是简单把请求体 parse 一下就结束了。`HttpServer` 里有连接级循环，`Router` 里有明确的匹配优先级，`ServletDispatch` 里有 middleware、pre/post interceptor 和 handler 的执行顺序及异常收尾语义，`HttpSession` 还区分了普通响应和流式响应的发送策略。所以这层本身就是项目的重要组成部分，不是陪衬。

**Q：你能把 HTTP 框架里一条连接从接入到断开，完整讲一遍吗？**

A：
可以，这一层我一般按“连接级生命周期”来讲。

第一步，监听 socket 在 `accept_worker` 上跑 accept 协程。底层 `TcpServer::startAccept()` 接到新连接后，不自己处理业务，而是把 `handleClient(client)` 调度到 `io_worker` 上执行。也就是说，accept 和实际请求处理一开始就被分开了。

这里最新一版和之前还有一个差别：它不是 accept 到连接后就随便挑个 io 线程，而是先从 `io_worker` 拿一份 worker 负载快照，再结合每个线程当前 inflight client 数做打分，分数公式大致可以理解成 `inflight * 8 + queuedTasks`。选出目标线程后，连接会被明确投递到这个 thread id 上，而不是只投给某个抽象 worker。

第二步，真正进入 `handleClient()` 之前，`TcpServer` 还会先把 client fd 绑定到当前 io 线程，确保这个连接后续生命周期里的 IO 操作保持线程亲和；等连接处理结束，再把这个线程上的 inflight 计数减回去。也就是说，接入均衡和 fd 线程亲和是在进入 HTTP 层之前就已经建立好的。

第三步，`HttpServer::handleClient()` 接到这个 client 后，先做两件基础事：如果启用了 SSL，就先把 socket 包成 `SslSocket` 并完成握手；然后用一个 `ScopedActiveConnection` 统计当前活跃连接数，如果超出上限，就尽量回一个 `503 Service Unavailable` 再断开。

第四步，真正进入连接级请求循环。这里不是“一连接一请求”，而是一个 while 循环反复做：

1. 根据是不是首个请求，设置不同的接收超时。第一次读请求走连接超时，后续 keep-alive 请求走 keep-alive 超时。
2. 调 `HttpSession::recvRequest()` 读取并解析一条 HTTP 请求。
3. 如果解析失败，区分错误类型回 `400 / 413 / 501`，然后结束连接。
4. 如果解析成功，就创建 `HttpResponse`，默认让响应版本跟随请求版本，keep-alive 语义也先跟随请求。
5. 如果 Session 功能开启，就先通过 `SessionManager::getOrCreate()` 处理 `SID` Cookie。
6. 再进入 `ServletDispatch` 做中间件、路由匹配和业务处理。
7. 如果是普通响应，`HttpSession::sendResponse()` 会一次性序列化 header + body 发出去；如果是流式响应，就只先发 header，body 交给业务层自己持续写。
8. 最后根据请求和响应双方是否都允许 keep-alive，以及是否达到 `keep-alive max requests`，决定是继续读下一条请求，还是关闭连接。

第五步，退出循环后，`HttpServer` 会显式 `session->close()`，这个连接生命周期才算结束。

所以 HTTP 层这里的关键点是：它处理的不是“单个请求函数调用”，而是“一个连接上的多次请求复用”。这也是为什么我觉得这层在面试里是值得讲的，因为很多人只会讲 handler，不会讲连接级状态机。

**Q：你的路由系统为什么要设计成精确匹配、参数路由、通配路由、默认路由这四层，而且优先级固定成 exact > param > glob > default？**

A：
因为路由匹配最怕的不是“功能少”，而是规则不稳定、不可解释。我的设计目标就是让它的命中结果可预测。

先看四层结构本身：

- 精确匹配，适合固定业务接口；
- 参数路由，适合像 `/conversations/:id` 这种路径；
- 通配路由，适合统一前缀或静态兜底；
- 默认路由，负责处理完全未命中的请求。

再看优先级为什么这么排。

第一，精确匹配必须最高。因为一个明确注册的固定接口，不应该被更宽泛的参数或通配规则抢走。如果 `/api/chat` 都可能被 `/:module/:action` 抢掉，整个路由系统就会变得不可控。

第二，参数路由要压过通配路由。因为参数路由虽然不是固定字符串，但它的结构仍然是明确的，是“限定段数和位置的模式匹配”；而通配往往只是前缀或 `*`，语义更宽。

第三，默认路由必须最后。它的存在不是参与竞争，而是为了统一处理未命中场景。

从实现上看，这个顺序也被明确写死在 `Router::match()` 里：先扫 `m_exact`，再扫 `m_params`，再扫 `m_globs`，最后回落到 `m_default`。参数路由还提前把 pattern 拆成 segments，并且只在真正命中时才把参数写回 request，这样匹配逻辑和副作用边界都比较清楚。

所以我会把这个设计总结成一句话：路由优先级的本质不是“哪个好实现”，而是“越具体的规则优先级越高”，这样才适合作为框架层长期维护。

**Q：为什么你的 `ServletDispatch` 不只是“路由 + handler”，而是还要分 middleware、pre interceptor、post interceptor？这套执行语义是怎么设计的？**

A：
因为 HTTP 框架里有几类逻辑其实不是一回事，混在一起会让边界很乱。

第一类是 middleware，它处理的是“和具体业务正交，但又必须覆盖整条请求链”的东西，比如请求 ID、鉴权、统一链路信息。这一层应该尽量靠前，而且允许短路。

第二类是 pre interceptor，它更像进入某类业务 handler 之前的局部拦截点，语义比 middleware 更贴近具体业务链路。

第三类是 post interceptor，它负责在 handler 后做补充处理或收尾，但它和 middleware 的 `after` 也不完全一样。

当前实现里，`ServletDispatch::handle()` 的主逻辑是：

1. 先跑 middleware before，任意一个返回 false 就短路主流程。
2. 如果没被短路，再按顺序跑 pre interceptor，任意一个返回 false 也会短路 handler。
3. 只有前置链路都通过了，才去做路由匹配和 handler 调用。
4. 正常路径下，如果进入了 post 语义，就按注册顺序执行所有 post interceptor。
5. 如果进入了 middleware 流程，就按逆序执行 middleware after。
6. 如果中间任意地方抛异常，也会先执行 post/after 收尾，再把异常继续抛给上层。

这里最有意思的点有两个。

第一，`after` 必须逆序，因为它更像资源入栈后的出栈收尾。谁先做 before，谁就应该后做 after，这样语义最对称。

第二，异常路径也必须收尾。否则你就会得到“正常请求链路没问题，但一旦某个 handler 抛异常，中间件后置逻辑就不执行”的脏状态。现在这套 try/catch 包裹就是为了保证框架层的执行语义在异常场景下仍然完整。

所以 `ServletDispatch` 的价值不只是“帮你找到 handler”，而是在定义整个请求处理链条的控制流语义。

**Q：为什么你的 HTTP 层要做 Session，而且还做成滑动过期 + 定时清理？**

A：
因为 Session 这件事在服务端框架里本质上是在解决“连接无状态，业务有状态”这个矛盾。

HTTP 请求本身是离散的，但业务经常需要把同一个用户的多次请求关联起来。我的实现里，`SessionManager::getOrCreate()` 会先尝试从请求 Cookie 里拿 `SID`，如果命中有效会话就直接返回；如果没命中或已过期，就创建新 Session，并通过 `Set-Cookie` 把新的 `SID` 下发给客户端。

这里为什么用滑动过期，而不是固定创建时间后硬过期？因为大部分服务端 Session 的真实需求都是“只要用户持续活跃，就不应该频繁失效”。所以 `get(id)` 命中一个有效 Session 后，会先 `touch()` 更新最后访问时间，再存回存储层。这样会话寿命是随着访问滚动延长的。

再看清理策略，为什么不是每次请求顺手扫一遍过期会话？因为那会把会话管理的维护成本直接塞进主请求路径里。当前实现里是单独起了一个 sweep timer，周期性调用 `sweepExpired()` 清理过期 Session，而且这个定时器是 condition timer，绑定 `weak_ptr`，这样 `SessionManager` 自己销毁后，定时回调也不会把它强行吊住。

所以这层设计要表达的点是：Session 不是单纯“生成个随机字符串”，而是在框架层明确会话创建、续期、过期和清理的职责分层。

**Q：你的 HTTP 层为什么要把 keep-alive 做成连接级循环，而不是每次请求单独建连处理？**

A：
因为在真实 HTTP 服务里，连接和请求不是一一对应的。只要双方都允许 keep-alive，一个 TCP 连接上就可能连续承载多次请求。

所以 `HttpServer::handleClient()` 的主体是一个 while 循环，而不是“读一次、处理一次、立刻断开”。这里的设计要点有四个。

第一，首包和后续包的超时不同。第一个请求更多是在等客户端把连接真正发起并发来完整请求，后续请求则属于 keep-alive 复用阶段，超时策略应该更收敛。

第二，响应端也可以主动终止 keep-alive。比如达到 `keep-alive max requests` 了，就算请求端还想复用，服务端也可以把 `response->setKeepAlive(false)`，主动结束这个连接。

第三，框架会补 `Keep-Alive` 头，把 `timeout` 和 `max` 这些约束写回去，这样客户端知道这个连接还能怎么复用。

第四，解析失败、发送失败、客户端断开这几类异常路径都应该直接退出循环，而不是让连接停留在半死不活状态。

这套设计的本质，是把“连接是连接，请求是请求”这两个层次分开。很多同学写 HTTP 服务只会写 handler，但 keep-alive 真正考验的是你有没有把连接级语义想清楚。

**Q：为什么流式响应要专门做 `response->isStream()` 这条分支，而且 `HttpSession` 在流式模式下只发 header，不直接发完整响应？**

A：
因为流式响应和普通响应在发送语义上根本不是一回事。

普通响应的特点是：header 和 body 在发送前都已经完整确定了。所以 `HttpSession::sendResponse()` 对普通响应的处理很直接，直接把整个 response 串成一个完整字符串，一次性写出去。这样可以减少“header 发了但 body 没发完”的中间态风险。

但流式响应不是这样。像 SSE 这种场景，业务层拿到请求后，只能先确定响应头，比如 `Content-Type: text/event-stream`、`Connection: close`、`Cache-Control: no-cache` 这些；真正的 body 是后续一点点生成、一点点写出的。这个时候如果还要求 `HttpSession` 一次性把完整 response 发出去，语义上根本不成立。

所以我这里专门用了 `response->isStream()` 这条分支：

- 如果是普通响应，`sendResponse()` 发完整报文；
- 如果是流式响应，`sendResponse()` 只发 header，后续 body 由业务层自己通过 session 持续写。

这样 HTTP 框架层和业务层的职责就清楚了：框架负责把连接和响应模式准备好，业务层负责在这个连接上持续产出流式数据。

对 SSE 来说，`SSEWriter::sendEvent()` 又进一步把一个事件格式化成 `event / id / retry / data` 这些字段，再通过 `writeFixSize()` 写到连接上。也就是说，这里不是“为了支持 AI 所以硬塞个流式接口”，而是在 HTTP 框架层明确区分了普通响应和持续输出响应这两种协议语义。

**Q：为什么你觉得 accept/io 分离在 HTTP 层也值得讲，而不是只属于底层 TcpServer？**

A：
因为 HTTP 服务模型本身就是建立在这个拓扑上的。

如果 accept 和请求处理混在一个执行上下文里，那么连接接入、SSL 握手、请求解析、业务处理、慢请求拖住 worker 这些问题都会糊在一起。而现在的做法是：`TcpServer` 启动时就明确要求 `accept_worker` 和 `io_worker` 是不同 `IOManager` 实例，监听 socket 上来的新连接先由 `accept_worker` 接住，再按 `inflight + queuedTasks` 选目标 io 线程，把 client 丢给那个具体 thread id 跑 `handleClient()`，并在进入处理前绑定 fd 线程亲和。

这样一来，HTTP 层能获得两个好处。

第一，职责更清楚。accept 线程只负责接入，不负责业务；io 线程只负责连接生命周期和请求处理。

第二，拓扑更稳定。就算业务层有慢请求、长连接、SSE 流式输出，它影响的是 io worker，不会反过来把 accept 路径也拖慢；而且连接接入不是粗暴 RR，而是做了连接感知分配，避免某个 io 线程既挂着很多存量连接，又继续被塞新连接。

所以我在面试里一般不会把这件事孤立地讲成“底层 Reactor 模型”，而会直接说：HTTP 服务之所以能清楚地区分连接接入和请求处理，就是因为底层 `TcpServer` 已经把 accept/io 分离拓扑固化了，上层 HTTP 框架是在这个模型上运转的。

### 3. AI 应用层深挖问答

**Q：为什么你说这个项目的 AI 应用层不是“模型接口转发器”，而是“服务编排层”？**

A：
因为从源码看，`AI` 这一层根本不是“收到请求后把 JSON 原样转发给上游模型，再把结果原样吐回来”。

真正的主流程在 `ChatService` 里，它把一次聊天请求拆成了多段编排动作：先做 `sid / message / 依赖` 校验，再生成或复用 `conversation_id`，接着通过 `EnsureContextLoaded()` 把历史上下文和摘要预热进内存，然后调用 `LlmRouter` 决定这次到底走哪个 provider 和 model，之后还会做上下文预算裁剪、可选 `RAG` 召回、模型调用、消息异步持久化、内存上下文更新，以及必要时的摘要刷新。

也就是说，这一层真正负责的是“把一个聊天服务需要的子系统串起来”，而不是只负责“调用模型”。它同时在管：

- 身份隔离：请求里的主体是谁；
- 会话状态：历史消息和摘要怎么组织；
- 模型选择：这次到底打到哪个 provider；
- 结果返回：同步返回还是 `SSE` 流式返回；
- 数据落地：消息先怎么入队，后面怎么写库；
- 检索增强：什么时候需要 `RAG`，失败了要不要降级；

所以我会把它定义成服务编排层，而不是接口转发层。接口转发层只关心协议适配，但这里明显已经在做请求控制流、状态管理和异步链路解耦了。

**Q：为什么模型路由顺序要设计成“显式 provider > model 映射 > 默认 provider”？**

A：
这其实是在解决“请求意图优先级”问题。

`llm_router.cc` 里的 `Route()` 逻辑很明确。第一优先级是请求里显式给的 `provider`。这代表调用方已经明确指定“我要走哪个上游实例”，那框架就不应该再替它做猜测。第二优先级是 `model -> provider` 映射，也就是请求没写 provider，但写了具体 model，这时由服务端配置决定这个 model 归属哪个 provider。第三优先级才是默认 provider，作为最终兜底。

这个顺序的价值在于三点。

第一，显式意图永远压过隐式推断。调用方既然明确写了 provider，就应该优先尊重。

第二，`model_map` 让“模型归属”成为配置问题，而不是把判断逻辑写死在业务代码里。以后你新增一个 provider，或者把某个 model 从一个 provider 切到另一个 provider，只改配置，不需要改聊天主流程。

第三，默认 provider 只做兜底，不参与抢占。否则配置一多，路由行为会变得不可预测。

另外，源码里路由完成后还会再做一次“最终 model 决定”：如果请求没带 model，就回落到 provider 的 `default_model`。这说明它路由的不是“某个抽象模型名字”，而是“一个具体可调用的 provider 条目 + 最终模型名”。这样业务层拿到的就是可直接执行的结果，而不是还得再补一轮判断。

**Q：为什么鉴权中间件不把 token 校验逻辑散落到每个 handler，而是统一注入 `X-Principal-Sid`？**

A：
因为这不是为了少写几行代码，而是为了把“认证”和“业务身份语义”拆开。

`auth_middleware.cc` 的做法是：先判断是否公共路径；非公共路径再从 `Authorization` 里取 `Bearer Token`；然后调用 `AuthService::AuthenticateBearerToken()` 做真正校验；校验成功后，不是把 token 明文继续往下传，而是把一组统一身份信息写进请求头，包括 `X-Auth-User-Id`、`X-Auth-Username` 和最关键的 `X-Principal-Sid`。

这样设计的核心好处有三个。

第一，业务层不依赖具体鉴权方式。`AiHttpHandlers::BuildChatRequest()` 只认 `X-Principal-Sid`，不关心底层到底是 Bearer Token、Cookie，还是以后换成别的认证手段。这样聊天、历史查询这些 handler 拿到的是统一身份语义，不是认证细节。

第二，避免把 token 校验散落到每个接口。否则每个 handler 都得重复写“解析 header、查库、判过期、判撤销、判用户状态”这一套，不光重复，而且非常容易出现某个接口忘记校验、或者不同接口校验规则不一致的问题。

第三，后续的数据隔离边界也统一了。`ChatService`、历史查询、`RAG` 检索这些地方，最终依赖的都是 `sid`，所以中间件直接把“认证通过后的主体身份”规整成 `principal_sid`，下游就可以稳定复用这一套隔离键。

所以这里不是简单的“中间件里做鉴权比较优雅”，而是把认证结果抽象成业务层真正需要的身份主键。

**Q：为什么登录只保存 `token_hash`，不保存 token 明文？为什么主体 SID 又设计成 `u:<user_id>` 这种格式？**

A：
这两个设计其实都在服务“最小暴露原则”。

先说 `token_hash`。`auth_service.cc` 里的登录流程是先生成 access token 明文，再立刻做 `Sha256Hex(access_token)`，最后只把哈希值和过期时间写进仓储层。后续鉴权时，也是对请求里的 token 再做一次同样的哈希，然后按哈希查库。

这样做的意义很直接：数据库里不保存可直接拿来冒用的 token 明文。就算库被误读，攻击者也拿不到现成 bearer token；而服务端仍然能完成“是否存在、是否过期、是否撤销”的判定。这和密码只存哈希是同一个思路，只不过 token 生命周期通常更短。

再说 `principal_sid = "u:<user_id>"`。这个设计看起来简单，但很实用。它把“主体”从具体用户表结构里抽象成了统一字符串主键。当前是用户，所以前缀用 `u:`；以后如果你想扩展成团队、机器人、租户，其实可以继续沿用这套前缀语义，而不是把整个系统写死在一个裸 `user_id` 上。

另外，对聊天和 `RAG` 这类模块来说，它们只需要一个稳定的隔离主键，不需要知道鉴权表有多少列。所以 `u:<id>` 这种字符串 SID 天然更适合作为跨模块边界值，而不是到处传数值型用户主键。

**Q：为什么 `ChatService` 不能简单地把历史消息全塞给模型，而必须做上下文缓存、预算裁剪和摘要刷新？**

A：
因为真实聊天服务里，“能不能把 prompt 组织好”本身就是核心能力，不能把它当成模型客户端顺手做的事。

从 `chat_service.cc` 看，这里至少做了四层控制。

第一层，是上下文加载策略。`EnsureContextLoaded()` 不是每次请求都全量查库，而是第一次按 `sid#conversation_id` 把 recent 消息和摘要拉到内存里，后面优先走内存快照。这样控制的是热路径延迟。

第二层，是上下文窗口控制。`AppendContextMessages()` 会把内存上下文限制在 `max_context_messages` 以内，防止会话越聊越久时内存无限膨胀。

第三层，是 prompt 预算裁剪。`BuildBudgetedContextMessages()` 里先把摘要封成一条 `system message`，再把 recent 和 `RAG recall` 合并成候选集，然后显式为“当前这条 user 消息”预留 token 预算。剩余预算里，优先尝试保留摘要，再从后往前挑最近消息。这里有一个关键点：不是“历史越多越好”，而是“信息密度越高、离当前越近的内容优先级越高”。

第四层，是摘要刷新。上下文一旦累积到一定规模，就不能永远靠 recent messages 硬撑，否则 prompt 体积会一直涨。`MaybeRefreshSummary()` 会把旧摘要和待压缩的旧消息一起喂给模型生成新摘要，然后只保留 recent 窗口里的消息，把更老的内容折叠进摘要里。

所以这里解决的问题不是“怎么把历史拼接成字符串”，而是怎么在成本、延迟和上下文保真之间做平衡。这恰恰是 AI 服务端最容易被忽略、但面试里很值得深挖的一点。

**Q：为什么摘要刷新是按 token 阈值触发，而不是按对话轮数触发？**

A：
因为真正影响模型成本和上下文窗口占用的，是 token 体积，不是轮数。

`MaybeRefreshSummary()` 的做法很直接：先统计当前摘要和历史消息的估算 token 总量，只有超过 `summary_trigger_tokens` 才触发摘要；否则就算轮数很多，但每轮都很短，也没必要总结。

如果你按“固定轮数”触发，会有两个明显问题。

第一，不同轮消息长度差异很大。有的十轮寒暄可能还没一轮长代码或长文档消耗的 token 多。按轮数触发，会把一个真正和成本强相关的问题，错误地代理成了一个弱相关指标。

第二，轮数触发很难和 `max_context_tokens` 联动。现在这套实现里，预算裁剪和摘要刷新都是围绕 token 预算在工作，所以它们的控制指标是一致的：一个负责“本次 prompt 里能装多少”，一个负责“什么时候该把旧上下文压缩掉”。

另外，源码里还专门做了 `keep_recent`。也就是说，触发摘要后不是把所有旧消息都压成摘要，而是保留最近一段原始消息，把更早那部分折叠掉。这样做的意义是：模型仍然能直接看到最近对话的原始语气和局部上下文，而不是所有历史都变成二手摘要。

所以按 token 触发，本质上是在对齐模型上下文窗口这个真实约束；按轮数触发则更像一个拍脑袋阈值。

**Q：为什么流式聊天不是每收到一个 `delta` 就立刻持久化，而是先组装完整回复，最后再统一落库？**

A：
因为流式传输的目标是“尽快把生成中的内容推给前端”，而不是“把每个碎片都当成独立消息保存”。

`StreamComplete()` 里这条链路很清楚。它先发 `start` 事件，然后在流式回调里把每个 `delta` 一边追加到 `assembled`，一边通过 `emit("delta", ...)` 推给 `SSE` 客户端。只有当整个流成功结束后，才把最终组装好的 assistant 内容和 user 消息一起封成两条 `PersistMessage`，再统一进入 `PersistMessage()`。

如果按 delta 持久化，会有几个问题。

第一，存储语义会被打碎。数据库里你到底存的是“一个完整 assistant 回复”，还是几十上百个碎片？如果是后者，历史回放、上下文加载和 `RAG` 索引都会变复杂。

第二，持久化压力会放大。模型每吐一个 chunk 都触发一次入队甚至写库，这会把流式输出的高频事件直接传导到后端存储层。

第三，失败恢复很难定义。比如流式中途断了，前面已经落了几十条 delta，后面没了，这种半成品该怎么算一条消息？现在这套实现把“网络流式输出”和“最终消息落地”拆开后，存储层看到的始终是消息级对象，而不是 token 级碎片。

所以流式链路里，`start/delta/error/done` 这些事件是面向客户端交互的；而持久化仍然坚持消息级边界。这个分层是合理的。

**Q：为什么消息持久化要做成 `MessageSink` 抽象，而且优先走 RabbitMQ，失败再回退本地异步 MySQL Writer？**

A：
因为聊天主链路最怕的就是“模型已经生成完了，但请求线程还被落库动作拖住”。`MessageSink` 抽象就是在切这个耦合。

从 `main.cc` 看，启动时会根据配置装配不同 sink。优先尝试 `RabbitMqMessageSink`；如果 `MQ` 启动失败且配置允许，就回退到本地 `AsyncMySqlWriter`；如果两条都起不来，服务直接启动失败。

这背后的设计逻辑是分层的。

第一层，`ChatService` 根本不关心消息最终怎么落地，它只调用统一的 `Enqueue()`。这让业务主流程只依赖“异步写接口”，而不是直接依赖数据库或 `MQ` 细节。

第二层，`RabbitMqMessageSink` 和 `AsyncMySqlWriter` 都是“请求线程只入本地内存队列，后台线程慢慢刷出去”的模型。也就是说，不管后端是 `MQ` 还是 MySQL，请求线程都不会直接去做重 IO。

第三层，为什么优先 `MQ`？因为 `MQ` 天生更适合把“在线请求生成”和“消息持久化/后续消费”拆成两条链。请求线程只要把消息可靠投进生产者本地队列，再由后台发布线程发到 RabbitMQ，后面消费者怎么落库、怎么扩展就更灵活。

第四层，为什么还要保留本地异步 MySQL Writer？因为对个人项目或者单机部署场景来说，`MQ` 不是总能稳定提供，所以需要一个降级通路，避免整个系统因为外部中间件没起好就完全不可用。

所以这里体现的不是“我会接 RabbitMQ”，而是你把写路径抽成了一个稳定接口，再在接口后面挂不同的异步落地实现。

**Q：你这个 AI 服务里，哪些失败是必须中断主流程的，哪些失败是可以降级放过的？为什么？**

A：
这个问题很关键，因为它能看出你有没有区分“主功能”和“增强能力”。

从 `ChatService` 的实现看，主链路上的关键失败基本都是 `fail-close`，也就是直接让请求失败返回。比如：

- `sid`、`message` 这种基础参数不合法；
- `LlmRouter` 路由失败，说明连该打哪个模型都定不下来；
- `route.client->Complete/StreamComplete` 调用失败，说明核心回答能力失败；
- `m_sink->Enqueue()` 失败，说明消息连最基本的持久化通道都进不去；

这些失败之所以必须中断，是因为它们直接影响“这次回答是否成立”或者“这次消息是否被系统正式接收”。

但也有一些能力是明确做成 `fail-open` 的。最典型的两个：

- `MaybeRefreshSummary()` 失败只记日志，不阻断本次回答；
- `m_rag_indexer->Enqueue()` 失败只告警，不影响主流程返回；

原因很简单。摘要刷新是为了长期上下文压缩，`RAG` 索引入队是为了后续可检索性，它们都属于增强链路，而不是当前这次回答成立的前提条件。就算它们挂了，当前回答仍然可以交付，所以更合理的策略是放行请求、记录告警，而不是把增强功能失败升级成主功能失败。

另外，`RAG` 查询链路本身也做了降级：`BuildRagMemoryMessages()` 里如果检索失败，只是打 `WARN` 然后返回空记忆，不会把整次聊天打死。这说明系统在设计上已经把“回答主链路”和“附加增强链路”分开了。

**Q：为什么 RAG 要拆成“查询链路”和“索引链路”？为什么一个在主流程里同步等待，另一个放到后台线程异步执行？**

A：
因为这两条链路虽然都属于 `RAG`，但职责完全不同。

查询链路解决的是“当前这一问要不要用历史记忆增强回答”。所以 `ChatService::BuildRagMemoryMessages()` 里如果命中召回条件，就必须立刻去做一次 `query -> embedding -> vector search`，拿到结果后才能把检索片段拼进 prompt。也就是说，查询链路天然属于主请求流程，它的结果会直接影响当前回答内容，所以当前请求必须等待这一步完成。

索引链路解决的是“当前这条新消息以后能不能被检索到”。这件事不影响本次回答能不能生成，所以 `PersistMessage()` 里只是顺手把消息投给 `RagIndexer::Enqueue()`。真正的 embedding 和向量库 `upsert` 发生在 `RagIndexer` 自己的常驻后台线程里，它有自己的队列、批量抓取、`flush_interval` 和去重逻辑，不是每来一条消息就临时新开一个线程。

这种拆法的本质，是把“读时增强”和“写时建索引”分离。前者是在线主路径，要求结果立刻可用；后者是后台维护路径，更关注吞吐、批处理和去重。

源码里还能看到一个很实际的取舍：索引链路即使入队失败也只是告警，因为它是增强能力；查询链路如果真的执行了，就会在当前 fiber 里等待检索结果，因为这一步会影响 prompt 组装。这种差异化处理是合理的。

**Q：既然底层已经有 Hook 机制了，为什么 embedding 和向量库的 HTTP 调用还要单独做 `FiberCurlSession`，而不是直接靠 Hook 自动协程化？**

A：
因为这里用的不是普通 `socket/read/write` 调用链，而是 `libcurl easy` 这套内部状态机，单靠 Hook 看不穿它。

底层 Hook 的强项是接管当前线程里那些显式暴露出来的阻塞系统调用，比如 `connect`、`read`、`write`。但 `libcurl easy` 把一次 HTTP 请求封装在它自己的阻塞执行过程里，外层只看到一个 `curl_easy_perform()`。对于这种库内部自己管理 socket 和等待逻辑的情况，Hook 没法像接管原生 socket API 那样自然地把等待语义改写掉。

所以这里的做法是单独实现 `FiberCurlSession`：如果当前线程存在 `IOManager` 和当前 fiber，就走 `curl_multi + socket 回调 + timer 回调 + Fiber::YieldToHold()` 这套协程化状态机；如果不在协程环境里，再回退到阻塞式 `curl_easy_perform()`。

这个实现里还有一个很值得讲的细节：`sylar` 的 `IOManager` 事件是一次性触发语义，而 `libcurl multi` 的 socket 关注集合不是天然按这个模型设计的，所以 `FiberCurlSession` 在收到一次 fd 事件后，要主动重挂对应监听。这说明它不是简单“把 libcurl 套进协程”而已，而是真的把两套不同的事件语义适配到一起了。

所以这个点在面试里很好讲，因为它能说明你理解 Hook 的边界，也理解为什么第三方网络库有时候必须单独做协程适配层。
